# Dev Log — 2025-12-20

## PWA и Offline-режим для Telegram WebApp
- **Проблема:** Приложение не открывалось без интернета в Telegram WebApp, даже если ранее было загружено.
- **Причина 1:** Telegram добавляет динамический параметр `tgWebAppData` к URL, который меняется при каждом открытии. Service Worker (SW) не находил точного совпадения в кэше.
- **Причина 2:** Inertia.js загружает страницу динамически, и при первой установке SW (если пользователь просто зашел) HTML-страница не всегда попадала в кэш.
- **Решение:**

### 1. Service Worker (v6)
- **Файл:** `public/sw.js`
- **Игнорирование параметров:** При поиске в кэше теперь используется опция `{ ignoreSearch: true }`, чтобы игнорировать `?tgWebAppData=...`.
- **Кэширование "чистых" URL:** При сохранении страницы в кэш мы теперь сохраняем две копии:
  1. Оригинальный запрос (с параметрами).
  2. "Чистый" URL без параметров (например, `/master/calendar`).
- **Fallback:** Если страница не найдена, SW пытается отдать `/master/calendar` (главную точку входа) или главную страницу.
- **Cache Priming Support:** Добавлена обработка `fetch(event.request)` для HTML-запросов, инициированных вручную (через JS), чтобы они корректно кэшировались как навигационные запросы.

### 2. Принудительный прогрев кэша (Cache Priming)
- **Файл:** `resources/js/app.js`
- **Логика:** При регистрации SW и его активации (`controllerchange`) скрипт автоматически запускает функцию `primeCache()`.
- **Действие:**
  - Скачивает текущий HTML (через `fetch(window.location.href)`).
  - Скачивает все подключенные JS-скрипты и CSS-стили.
  - Это гарантирует, что сразу после первого открытия (с интернетом) все необходимые файлы попадут в кэш.

### 3. Manifest
- **Файл:** `public/manifest.json`
- **Уточнение:** Убедились, что `start_url` указывает на `/master/calendar`.

## Инструкция для проверки
1. Открыть приложение с включенным интернетом.
2. Подождать 3-5 секунд (для завершения прогрева кэша).
3. Закрыть приложение, отключить интернет.
4. Открыть приложение — оно должно загрузиться из кэша.
